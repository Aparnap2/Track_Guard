# Langfuse Trace Reading — Operating Procedure

> **Context:** Per the Anthropic agent talk, the best harness improvement loop is to manually read transcripts, find where agent judgment diverges from human standards, and tighten the contract at that point.
>
> Langfuse is already instrumented at 10 tests passing. This document defines the operational workflow for post-launch week 1.

---

## When to Run This Procedure

| Phase | Frequency | Trigger |
|-------|-----------|---------|
| **Week 1 post-launch** | Every alert that fires | First real tenant data |
| **Week 2-4** | Daily review of previous day's alerts | Batch mode |
| **After any code change** | Spot-check 3 alerts | Deployment to tenant |
| **Steady state** | Weekly batch | Monday morning |

---

## Step-by-Step: Read a Langfuse Trace

### Step 1 — Open Langfuse

1. Navigate to your Langfuse instance:
   - Self-hosted: `http://langfuse.local`
   - Cloud: `https://cloud.langfuse.com`
2. Filter by:
   - **Trace name:** `sarthi-agent-run`
   - **Time range:** Last 24h (week 1) or last 7d (steady state)
   - **Tags:** `tenant_id:<your_tenant>`

### Step 2 — Identify a Divergence

For each trace, read the full LLM call chain. Look for the specific **LLM call** where the model's judgment diverges from what a human engineer would decide.

Common divergence patterns:

| Pattern | What to Look For | Fix Location |
|---------|-----------------|--------------|
| **Over-alerting** | LLM decides `should_alert: true` for a pattern that is clearly a healthy fluctuation | [`src/agents/bi/nodes.py`](apps/ai/src/agents/bi/nodes.py) — tighten cognitive decision prompt |
| **Under-alerting** | LLM decides `should_alert: false` despite rule engine flagging it | [`src/agents/finance/graph.py`](apps/ai/src/agents/finance/graph.py) — lower threshold or add guardrail |
| **Wrong severity** | LLM assigns "info" to a pattern that should be "critical" | [`src/schemas/guardian.py`](apps/ai/src/schemas/guardian.py) — tighten Pydantic validator |
| **Hallucinated number** | Narrative contains a number not found in the injected data | [`src/guardian/insight_builder.py`](apps/ai/src/guardian/insight_builder.py) — enforce Rule 1 (numbers NEVER generated by LLM) |
| **Wrong tone** | Alert reads like a notification, not like a trusted colleague | [`src/agents/bi/prompts.py`](apps/ai/src/agents/bi/prompts.py) — add tone constraint to system prompt |

### Step 3 — Record in Trace Reading Log

Use this format to log every trace you read:

```markdown
## Trace: {trace_id}
- **Alert ID:** {alert_id}
- **LLM Call ID:** {langfuse_generation_id}
- **Expected Outcome:** {what the human engineer would decide}
- **Actual Outcome:** {what the LLM decided}
- **Divergence:** {over_alerting | under_alerting | wrong_severity | hallucinated_number | wrong_tone}
- **Root Cause:** {the specific prompt or contract that needs tightening}
- **Fix Applied:**
  - File: `apps/ai/src/agents/bi/prompts.py` line 47
  - Change: Added "never alert on single-day fluctuations" to system prompt
  - PR: #142
- **Verification:** After fix, re-ran trace with same input. Outcome matches expected.
```

### Step 4 — Tighten the Contract

For each divergence, choose the appropriate fix:

| Divergence | Tightening Strategy | Example |
|------------|-------------------|---------|
| **Over-alerting** | Add negative example to the system prompt or Pydantic field validator | `"context_note must mention if trend is single-day or multi-day"` |
| **Under-alerting** | Add explicit rule in the cognitive decision prompt | `"if rule_anomalies contains 'burn_spike' AND runway < 180, must alert"` |
| **Wrong severity** | Tighten the Literal type or add a severity mapping rule | Map `runway_days < 90` → always critical, never info |
| **Hallucinated number** | Audit the narrative prompt — remove all "you may include" language | Make injected numbers mandatory, narrative optional |
| **Wrong tone** | Add a banned-phrase list to the narrative prompt | `"Never start with a number. Never use bullet points. Always start with pattern name."` |

### Step 5 — Verify

After applying the fix:

1. Find the original trace in Langfuse
2. Copy the exact input that caused the divergence
3. Run the updated agent with that same input (use `pytest` with the trace's raw data)
4. Verify the output now matches expected outcome
5. Update the Trace Reading Log with the verification result

## Exceptions — When NOT to Tighten

- **One-off anomaly:** A single weird output from a model you've seen perform well 100 times. Log it, don't fix it.
- **Prompt template is fine, model was derailed by context:** If the divergence was caused by unusual context, not the prompt, the fix is in context assembly, not the prompt.
- **Divergence is a feature, not a bug:** If the model correctly ignored a stale rule, the rule needs updating, not the prompt.
